## はじめに

Claude Codeには「スキル」という機能がある。`.claude/skills/` ディレクトリに `SKILL.md` というMarkdownファイルを置くだけで、Claude Codeに専門的な業務遂行能力を付与できる仕組みだ。

プロンプトテンプレートとは根本的に違う。スキルは **入力・手順・出力・品質基準** を一体で定義し、Claude Codeが自律的に業務を遂行するための「業務マニュアル」として機能する。

筆者は半年かけて30のスキルを開発・運用してきた。商品企画の市場調査から議事録作成、ブラウザ自動操作まで、日常業務のかなりの部分をスキル経由で回せるようになった。本記事ではその全貌と設計ノウハウを共有する。

## スキルの基本構造

### SKILL.mdのフォーマット

スキルファイルは極めてシンプルだ。YAMLフロントマター + Markdown本文で構成される。

```markdown
---
description: このスキルが何をするかの1行説明
allowed-tools: Read, Write, Edit, Glob, Bash
---

# スキル名

## コマンド
$ARGUMENTS

## 手順
1. ...
2. ...

## 出力
- ファイルパス: ...
- フォーマット: ...
```

`description` でClaude Codeがスキルの用途を判断し、`allowed-tools` で使用可能なツールを制限する。`$ARGUMENTS` にはユーザーが渡した引数が展開される。

### 実例: todoスキル（114行）

最もシンプルなスキルの1つが `todo` だ。タスク管理表（Excel）の確認・更新を行う。

```markdown
---
description: タスク管理表（Excel）の確認・更新・管理を行う
allowed-tools: Read, Write, Edit, Glob, Bash, AskUserQuestion, Agent
---

# タスク管理スキル

## コマンド
$ARGUMENTS

引数なしの場合は現在のタスク一覧を表示します。
引数がある場合はその指示に従って更新します。

## モード判定

入力内容に応じて以下のモードで動作します:

### MODE A: テキストステータス更新
短い指示でステータスを更新する。
- 例: `#3完了` → タスク#3のステータスを「9.完了」に変更

### MODE B: 議事録→タスク抽出（バルクインポート）
議事録ファイルからタスクを一括抽出する。
- 例: `/todo インポート docs/議事録/20260320_議事録.md`
```

ポイントは **モード判定** だ。入力パターンに応じて動作を切り替えることで、1つのスキルで複数のユースケースに対応する。`/todo #3完了` でステータス更新、`/todo` だけで一覧表示、引数に議事録パスを渡せばタスク抽出。自然言語コマンドのように使える。

## 最大のスキル: product-planning-orchestrator（582行）

30スキルの中核が `product-planning-orchestrator` だ。582行のSKILL.mdで、10以上のサブスキルを自律制御するメタオーケストレーターとして機能する。

### AI管理職というコンセプト

```markdown
## 役割: AI商品企画部長

あなたはアイリスオーヤマの **商品企画部長（AI管理職）** である。
テーマを受け取ったら、Gate 1〜3 の全ステップを **自律的に判断・実行** し、
高品質な商品企画パッケージを完成させる。

**基本原則**:
- すべての業務判断・品質判断は自分で行う
- CEOに確認するのは「方向性の核」と「最終承認」のみ
- 各ステップで品質ゲートコントローラー（QGC）を適用し、合格するまで次に進まない
```

ポイントは **CEOへの確認を4回に限定** していること。

1. **キックオフブリーフ** - 企画の方向性確認
2. **コンセプト承認** - Gate 1完了時
3. **事業計画承認** - Gate 2完了時
4. **最終パッケージ承認** - Gate 3完了時

それ以外の判断（調査の深さ、分析手法の選択、成果物のフォーマット）はすべてAIが自律判断する。

### スキル間のデータ受け渡し

サブスキル間の連携は **ファイルベース** で実現する。各スキルの出力は `_AI_knowledge/` ディレクトリに保存され、後続スキルがそこから読み取る。

```
workspace/iris-product/KANTERI_2026/
├── _AI_knowledge/
│   ├── market_research.json       # market-researchの出力
│   ├── customer_insight.json      # customer-insightの出力
│   ├── competitive_analysis.json  # competitive-analysisの出力
│   └── concept_decisions.json     # orchestratorの判断ログ
├── 成果物/
│   ├── コンセプトシート_KANTERI.xlsx
│   ├── PRD_KANTERI.md
│   └── 事業計画_KANTERI.xlsx
└── project_summary.json           # プロジェクト全体のサマリ
```

この設計により、各スキルは **独立して開発・テスト** できる。orchestratorを変更しても個別スキルに影響しないし、逆も然り。

### 品質ゲートコントローラー（QGC）

各ステップの出力には品質ゲートが設けられている。内部でCriticロールが品質を評価し、基準を満たさなければ最大3回まで自動リトライする。

```
社長の「で、いくら儲かるの？」「根拠は？」に耐えられるかが品質の基準
```

このように具体的な判断基準をSKILL.mdに書くことで、抽象的な「高品質」ではなく、**組織固有の品質水準** をAIに伝えられる。

## スキル設計のコツ

30スキルを開発する中で学んだ設計原則をまとめる。

### 1. 1スキル1責務

`market-research` と `customer-insight` を1つの「調査スキル」にまとめたくなるが、分けるべきだ。理由:

- **テスト可能性**: 単機能のほうが出力の品質を評価しやすい
- **再利用性**: 他のオーケストレーターからも個別に呼べる
- **保守性**: 修正の影響範囲が限定される

### 2. 入出力を明確に定義

```markdown
## 出力
- ファイル: `workspace/iris-product/{PJ}/成果物/競合分析_{PJ}.xlsx`
- フォーマット: Excel（シート1: 競合一覧、シート2: ポジショニングマップデータ）
- 併せて `_AI_knowledge/competitive_analysis.json` にサマリを保存
```

ファイルパスとフォーマットを明示することで、後続スキルが迷わずデータを取得できる。

### 3. ファイルベースの疎結合

スキル間の連携にAPIやメモリ共有は使わない。すべてファイルシステム経由。

- メリット: デバッグが容易。中間成果物を目で確認できる
- メリット: スキルの追加・削除が他に影響しない
- メリット: 人間が手動で中間ファイルを修正して再開できる

### 4. 品質ゲート（QGC）を組み込む

重要なスキルにはCriticパターンを入れる。

```markdown
## 品質ゲート
出力完了後、以下の観点でセルフレビューを行い、
不合格なら修正して再出力する（最大3イテレーション）:

1. 数値に根拠があるか（出典URLまたは算出ロジック）
2. 抽象的な表現が残っていないか
3. 指定フォーマットに準拠しているか
```

「最大3イテレーション」という上限が重要。無限ループを防ぎつつ、品質を担保する。

### 5. CEOタッチポイントを最小化

自律判断の基準を明文化することで、人間への確認回数を減らす。

```markdown
## 自律判断基準
以下の場合はCEO確認なしで判断してよい:
- 価格帯が市場相場の0.8〜1.2倍の範囲内
- ターゲット変更が当初仮説の周辺（年齢±10歳、同カテゴリ内）
- 追加調査が必要だが所要時間が30分以内
```

判断の「閾値」を数値で示すのがコツ。曖昧な「大きな変更はCEOに確認」ではなく、具体的な条件を書く。

## 実際の成果

### KANTERI_2026プロジェクト

`product-planning-orchestrator` を使い、新商品「KANTERI」の企画を実行した結果:

- **1日で15成果物を生成**: 市場調査レポート、顧客インサイト、競合分析、コンセプトシート、PRD、事業計画、キックオフチェックリストなど
- **ROI 3年74.5倍の事業計画**: 市場データに基づく根拠付きの収支シミュレーション
- **5部門32タスクのキックオフチェックリスト**: 開発・製造・品質・営業・マーケの初動タスクを自動生成

従来なら2-3週間かかる企画初期フェーズが、CEOの確認4回を含めて **実質1営業日** で完了した。

### スキルの成長サイクル

スキルは書いて終わりではない。実案件で使うたびにフィードバックが発生する。

1. スキルを実行 → 成果物を確認
2. 「数値の根拠が弱い」「フォーマットが違う」などの課題を発見
3. SKILL.mdに品質基準やフォーマット指定を追記
4. 次回実行で品質が向上

このサイクルを回すことで、30スキルは半年間で継続的に品質が上がり続けた。

## まとめ

スキルは「プロンプトの再利用」ではない。 **業務プロセスのコード化** だ。

- プロンプトテンプレート: 「こう聞けばいい回答が返る」
- スキル: 「この手順でこの品質の成果物を出す」という業務プロセス定義

30スキルの蓄積は、個人の生産性向上にとどまらない。SKILL.mdとしてGit管理されたスキル群は、**組織のAI活用能力そのもの** になる。新メンバーが入っても、スキルを実行すれば同じ品質の成果物が出る。属人化しない。

Claude Codeのスキル機能は、`.claude/skills/` にMarkdownファイルを置くだけ。特別なインフラもフレームワークも要らない。まずは日常で繰り返している業務を1つ選んで、SKILL.mdに書き起こすところから始めてみてほしい。
